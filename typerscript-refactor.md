# RSS2Pod TypeScript 重构宏观文档

## 一、项目概述

### 1.1 项目定位

RSS2Pod 是一个**个人化自动化系统**，将多源 RSS 信息经过以下处理流程，最终以标准 Podcast Feed 形式发布：

```
RSS 信息 → 结构化压缩 → 对话化重构 → 英语学习增强 → 音频化生成 → Podcast Feed 发布
```

### 1.2 核心价值

- **知识内化**：将被动阅读转为主动收听
- **英语学习**：在获取信息的同时学习英语
- **自动化**：无需人工干预，自动处理新内容
- **个性化**：可根据个人偏好定制处理策略

---

## 二、核心功能需求

### 2.1 内容采集层

#### 2.1.1 RSS 数据获取
- 通过 Fever API 协议与 RSS 阅读器（如 TT-RSS）交互
- 获取已订阅的文章列表和内容
- 周期性拉取但不标记为已读（保持原始状态）
- 将文章内容转换为纯文本（排除图片，减少 token 消耗）

#### 2.1.2 文章管理
- 本地缓存文章内容，减少重复 API 调用
- 存储文章元数据：标题、来源、发布时间、正文、归属源
- 追踪文章被哪些组处理过
- 支持文章导出功能

---

### 2.2 分组（Group）管理

#### 2.2.1 Group 概念
每个 Group 是一个**信息处理策略容器**，包含：
- 一组绑定的 RSS 订阅源
- 一套处理规则（Skills）
- 触发条件配置
- 播客结构定义（单人/双人）
- 独立的 Podcast Feed 输出

#### 2.2.2 Group 功能
- 创建、编辑、删除 Group
- 绑定/解绑 RSS 源
- 配置摘要生成倾向
- 配置英语学习增强模式
- 配置触发策略
- 配置音频保留周期
- 启用/禁用 Group

---

### 2.3 触发机制

系统支持三种触发方式，可组合使用：

#### 2.3.1 时间触发
- 基于 Cron 表达式
- 支持每日/每周/指定时间触发
- 到时间检测是否满足生成条件，无新内容可跳过

#### 2.3.2 数量触发
- 当未处理文章数量达到阈值时触发
- 立即进入生成流程

#### 2.3.3 LLM 判断触发
- 将未处理文章摘要交给 LLM 分析
- LLM 判断是否具有集中主题、生成价值
- 根据判断结果决定是否触发

---

### 2.4 内容处理层

#### 2.4.1 源级摘要（Source Summary）
- 同一 RSS 源的多篇文章拼接后统一处理
- 生成该源的摘要内容
- 摘要风格由配置决定（宏观分析/技术细节等）

#### 2.4.2 组级摘要（Group Summary）
- 将所有源级摘要聚合
- 生成组级总摘要
- 作为播客脚本生成的输入

**两层摘要的目的**：
- 控制 token 使用量
- 提升主题集中度
- 提高内容结构质量

---

### 2.5 播客脚本生成

#### 2.5.1 脚本引擎
- 抽象的脚本生成接口
- 支持多种实现方式（LLM 生成、模板引擎等）
- 可配置、可替换

#### 2.5.2 播客结构模式

**单人播报模式**：
- 类似新闻播音
- 结构紧凑
- 适合 Daily Brief

**双人对话模式**：
- 主持人 + 协主持人
- 自然交替对话
- 增强沉浸感

#### 2.5.3 英语学习增强
支持三种模式：
- 正常播报
- 句后难词解释
- 句前整句翻译

设计原则：
- 翻译自然
- 不打断逻辑流
- 不显得机械
- 不重复冗余

---

### 2.6 音频合成层

#### 2.6.1 TTS 抽象
- 统一的 TTS 接口设计
- 支持不同云 TTS 服务（如 SiliconFlow）
- 支持单人/双人语音配置
- 支持语速调节

#### 2.6.2 音频处理
- 分段 TTS 合成（解决 API 时长限制）
- 多段音频拼接为完整 MP3
- 保证音量统一、无明显断层
- 音频文件管理（存储、清理）

---

### 2.7 Podcast Feed 发布

#### 2.7.1 Feed 生成
- 每个 Group 生成独立的 Podcast Feed
- iTunes 兼容的标准 RSS 格式，可使用podcast npm库用于生成合法feed
- 包含音频 enclosure（支持 HTTP Range 请求）
- 嵌入 HTML 格式的原文对照

#### 2.7.2 Feed 管理
- Feed URL 固定且长期稳定
- 支持在播客客户端（如 AntennaPod）订阅
- 支持星标内容永久保留
- 支持按保留周期自动清理

---

### 2.8 内容清理机制

#### 2.8.1 保留策略
- 星标内容永久保留
- 非星标内容按保留周期删除

#### 2.8.2 清理流程
严格顺序执行：
1. 从 Feed 中移除条目
2. 删除音频文件
3. 更新数据库状态

- **可追踪**：记录所有事件的流转过程
- **可扩展**：方便新增事件类型和处理器

#### 3.2.3 事件数据结构

每个事件包含以下标准字段：

- **事件 ID** - 事件的唯一标识符
- **事件类型** - 标识事件的种类和用途
- **时间戳** - 事件发生的精确时间
- **载荷数据** - 事件携带的具体业务数据
- **事件来源** - 标识事件的发布来源
- **关联 ID** - 用于追踪完整事件链的标识
- **元数据** - 包含组 ID、用户 ID、优先级等附加信息

#### 3.2.4 事件处理器

事件处理器是系统中负责响应和处理事件的组件，具备以下能力：

- **事件处理能力** - 接收并处理订阅的事件
- **事件过滤能力** - 可选择性地处理特定事件
- **错误处理能力** - 处理事件处理过程中发生的异常
